# 工作流编辑与运行体验增强

## 任务背景
当前工作流页已经具备基础自动保存、节点拖拽、连线、运行和结果查看能力，但整体编辑体验仍然偏“功能可用”，和 Make 一类工作流编辑器相比，仍缺少一组关键编辑与运行辅助能力：

1. 保存反馈不够明确
当前页面会自动保存到服务端，但状态主要显示在左侧列表顶部，缺少贴近画布操作区的统一保存反馈，也缺少保存失败后的明确补救入口。

2. 缺少一键自动排列
当前节点首次进入画布时仅按默认纵向位置初始化，后续拖拽后无法快速恢复为结构化布局，分支稍多时画布很容易变乱。

3. 缺少流程解释视图
当前页面主要提供节点级详情和运行结果，没有一个从“触发器 -> 节点执行顺序 -> 入参与出参传递关系”角度解释整个集成流的视图，新用户理解成本较高。

4. 请求无法通过拖拽直接编排
当前页面已有可用请求列表，但仍以点击插入为主，缺少像 Make 一样从请求列表拖入画布进行编排的能力。

5. 缺少节点搜索定位
此前节点搜索入口已移除，当前工作流节点一旦变多，用户无法根据名称快速找到某个节点并定位到画布位置。

6. 运行结果只停留在当前会话
当前页面仅展示当前这次运行的结果，页面刷新后即丢失，也缺少历史运行记录列表，以及按一次运行展开查看各节点执行详情的能力。

## 任务目标
在不重做现有工作流编辑器的前提下，补齐以下能力：

1. 提供 Make 风格的底部操作条，统一承载保存反馈和画布辅助操作。
2. 提供一键自动排列能力，支持将当前工作流节点重新整理为可读布局。
3. 提供集成流解释面板，用自然语言加结构化摘要解释当前工作流的执行链路。
4. 提供请求列表入口，支持将请求拖拽到工作流画布中完成编排。
5. 提供节点搜索入口，支持根据节点名称快速查找并定位。
6. 提供运行日志中心，支持保存每次工作流运行结果并按节点回看执行详情。

## 需求范围

### 一、保存能力增强

#### 目标
让用户在画布编辑过程中，随时知道当前工作流是否已保存、正在保存、保存失败，且能在失败时快速重试。

#### 需求说明
1. 在工作流画布底部增加固定操作条，保存状态需放在操作条内展示，而不是仅放在左侧列表顶部。
2. 保存状态至少包含以下四种：
   - 未保存
   - 保存中
   - 已保存
   - 保存失败
3. 以下操作都应触发保存状态变化：
   - 工作流重命名
   - 节点新增、删除、复制
   - 节点拖拽移动
   - 连线新增、删除
   - 节点参数输入修改
   - 自动排列执行后的位置更新
4. 自动保存仍保留当前延迟保存机制，但需要对用户可见。
5. 当保存失败时，底部操作条中需要提供“重试保存”入口。
6. 点击“运行工作流”时，如果当前存在未完成的保存请求，需要优先等待保存完成；如果保存失败，则给出提示并阻止本次运行。

#### 交互要求
1. 底部操作条默认悬浮在画布底部居中位置，视觉上独立于右下角缩放控件。
2. 保存状态文本应简短明确，例如：
   - 未保存
   - 保存中...
   - 已保存
   - 保存失败，请重试
3. 保存成功后可短暂展示“已保存”，随后退化为更轻量的静态状态。

#### 非目标
1. 本次不做版本历史。
2. 本次不做多人协同编辑冲突处理。
3. 本次不做离线缓存与离线恢复。

### 二、自动排列能力

#### 目标
支持用户在工作流结构变复杂后，一键将节点恢复为清晰、稳定、可继续编辑的布局。

#### 需求说明
1. 在底部操作条中提供“自动排列”按钮。
2. 自动排列基于当前工作流的触发器、节点和连线关系重新计算 `nodePositions`。
3. 布局原则：
   - 触发器保持在顶部中心区域
   - 下游节点按照执行层级自上而下排列
   - 同层分支横向展开
   - 节点之间保持统一间距，避免重叠
   - 尽量减少连线交叉
4. 对于没有连线的节点，需要单独归类排列，避免压在主流程上。
5. 对于存在异常结构的工作流（如回环、缺失边、孤立节点），自动排列不能报错，需要给出可用的回退布局。
6. 自动排列完成后：
   - 立即刷新画布中的节点位置
   - 持久化到当前工作流
   - 触发保存流程
   - 将视角重置或聚焦到可完整查看主流程的位置

#### 非目标
1. 本次不做用户手工锁定节点位置。
2. 本次不做多种布局算法切换。
3. 本次不做复杂的路径正交路由器重构，优先在现有曲线连线基础上提升布局可读性。

### 三、流程解释能力

#### 目标
让用户不需要逐个点开节点，也能快速理解“这个工作流在做什么、按什么顺序执行、上下游之间如何传值”。

#### 需求说明
1. 在画布区域增加“流程解释”视图，建议放置在画布左侧，并支持展开/收起。
2. 流程解释视图需基于当前工作流配置自动生成，不依赖外部 AI 服务。
3. 流程解释至少包含以下信息：
   - 工作流名称
   - 触发方式（当前为手动触发器）
   - 主执行链路说明
   - 每个节点的名称、方法、目标地址
   - 每个节点的上游来源
   - 每个节点消费了哪些输入参数
   - 哪些输入来自上游节点输出
   - 哪些节点没有接入主链路
4. 展示形式应同时包含：
   - 概览摘要：用 1 到 3 段自然语言总结当前流程
   - 步骤列表：按执行顺序列出节点
5. 步骤列表中的每一项应支持与画布联动：
   - 点击步骤后聚焦对应节点
   - 当前选中节点时，在解释面板中高亮对应步骤
6. 当工作流为空、无连线或结构不完整时，解释区需要输出清晰提示，而不是空白。

#### 推荐展示内容
1. 概览摘要示例：
   - 该工作流由手动触发开始，先请求用户信息，再根据用户 ID 查询订单，最后汇总结果返回。
2. 步骤说明示例：
   - 第 1 步：请求 A，调用 `GET /users/{id}`，为后续节点提供 `userId`
   - 第 2 步：请求 B，调用 `GET /orders`，其中 `userId` 来自“请求 A / userId”

#### 非目标
1. 本次不做自然语言编辑回写工作流。
2. 本次不做大模型生成解释。
3. 本次不做运行后动态回放动画。

### 四、请求列表与拖拽编排

#### 目标
用户点击“请求”入口后，可以看到所有可用于工作流编排的请求列表，并通过拖拽的方式把请求加入到工作流画布中。

#### 需求说明
1. 在工作流页面提供“请求”入口，点击后展示请求列表面板。
2. 请求列表需展示当前用户可用于工作流的全部请求，优先复用现有 `availableRequests` 数据来源。
3. 请求项至少展示：
   - 请求名称
   - 请求方法
   - 来源信息（如公开请求所属用户）
4. 用户可将请求项从列表中拖拽到画布区域，完成节点新增。
5. 拖拽落点规则：
   - 拖到空白画布时，在落点附近创建节点
   - 拖到主流程附近时，允许按最近可用位置插入
   - 如果当前工作流为空，应默认落在触发器下方主流程起始位置
6. 拖拽新增节点后：
   - 自动选中新节点
   - 写入 `workflow.requests`
   - 更新 `nodePositions`
   - 触发保存流程
7. 第一版允许保留当前“点击添加”作为兼容入口，但拖拽应成为主要交互方式。

#### 交互要求
1. 请求列表面板应支持展开/收起，不长期遮挡主画布。
2. 拖拽过程中需提供明确的拖动态、可放置区域反馈和放置成功反馈。
3. 请求很多时，列表需要支持滚动查看。

#### 非目标
1. 本次不做跨页面拖拽。
2. 本次不做按文件夹树结构拖入工作流，第一版允许使用平铺列表。
3. 本次不做请求模板编辑能力。

### 五、节点搜索定位

#### 目标
用户点击“搜索”入口后，可以根据节点名称快速在当前工作流中找到目标节点并定位到画布位置。

#### 需求说明
1. 在工作流页面提供“搜索”入口，点击后展示节点搜索面板或搜索输入框。
2. 搜索范围限定为当前选中工作流内的节点。
3. 搜索方式以名称模糊匹配为主，第一版不要求支持方法、URL、标签等多字段联合搜索。
4. 搜索结果项至少展示：
   - 节点名称
   - 请求方法
   - 简短地址或补充信息
5. 点击搜索结果后，需要：
   - 将画布视角移动到目标节点
   - 选中该节点
   - 在搜索结果列表中高亮当前项
6. 当输入为空时，可展示当前工作流全部节点列表，便于快速跳转。
7. 当无匹配结果时，需要给出明确空态提示。

#### 交互要求
1. 搜索输入应支持连续输入实时过滤。
2. 画布中的匹配节点可选做高亮，至少需支持结果点击定位。
3. 搜索能力不能恢复为之前已删除的旧搜索形态，需要和新工作流界面统一设计。

#### 非目标
1. 本次不做全局跨工作流搜索。
2. 本次不做高级筛选。
3. 本次不做搜索历史。

### 六、运行日志与执行回看

#### 目标
每次工作流运行后，都能保存一次完整运行记录，并在“日志”中查看运行摘要和节点级执行详情。

#### 需求说明
1. 在工作流页面提供“日志”入口，点击后展示当前工作流的运行日志列表。
2. 每次工作流运行完成后，无论成功或失败，都应生成并保存一条运行日志。
3. 日志列表每条记录至少展示：
   - 运行时间
   - 运行状态
   - 运行耗时
4. 点击任一日志后，进入该次运行的详情视图。
5. 日志详情需至少包含两个区域，参考用户提供的示意图：
   - 左侧：本次运行中按顺序执行的节点列表，并展示每个节点的执行状态
   - 右侧：当前选中节点的执行详情
6. 节点执行详情需至少展示：
   - 此节点的实际入参内容
   - 此节点的实际响应内容
   - 节点运行时间
   - 节点运行状态
   - 节点运行耗时
7. 对于执行失败的节点，需要展示错误信息，并终止链路的状态说明。
8. 日志在页面刷新后仍应保留，可重新打开查看。
9. 日志详情展示应基于运行时快照，而不是依赖当前工作流最新配置，避免节点名称或参数在后续被修改后导致历史日志失真。

#### 交互要求
1. 日志列表默认按最新时间倒序展示。
2. 点击某条日志后，应默认选中该次运行中的第一个节点或第一个异常节点。
3. 节点列表与详情区域需要联动切换。
4. 如果当前工作流尚未运行过，需要显示“暂无日志”空态。

#### 数据要求
1. 运行日志需要保存工作流级摘要：
   - workflowId
   - workflowName
   - startedAt / finishedAt
   - durationMs
   - status
2. 运行日志需要保存节点级结果：
   - requestId
   - requestName
   - status
   - statusCode
   - durationMs
   - requestInfo
   - responseData 或 error
3. 如日志中包含敏感请求头、鉴权信息或 Cookie，展示和持久化时应做脱敏处理。

#### 非目标
1. 本次不做日志导出。
2. 本次不做日志全文检索。
3. 本次不做按时间范围筛选、分页优化之外的复杂日志分析能力。

## 页面与交互安排

### 1. 底部操作条
建议位于画布底部中间区域，参考 Make 的底部工具带，包含但不限于：
- 保存状态
- 重试保存
- 自动排列
- 视图辅助操作（如重置视角，可沿用现有能力整合）
- 流程解释面板展开/收起开关

### 2. 左侧辅助入口区
建议在工作流列表侧栏和主画布之间，增加一组轻量入口或抽屉式面板，至少包含：
- 请求
- 搜索
- 日志
- 流程解释

点击不同入口后，可在同一左侧面板区域切换对应内容。

要求：
- 不遮挡主流程核心区域
- 在小屏下允许折叠
- 与现有右侧节点详情、执行结果面板共存

### 3. 请求入口
“请求”入口打开后，展示可拖拽请求列表，用于向画布新增节点。

### 4. 搜索入口
“搜索”入口打开后，展示当前工作流节点搜索框和结果列表，用于快速定位节点。

### 5. 日志入口
“日志”入口打开后，展示当前工作流运行日志列表；点击日志后，进入该条日志的节点级回看视图。

### 6. 现有控件整合
当前右下角缩放控件建议保留，但应和新的底部操作条形成清晰分工：
- 底部操作条负责状态与高频动作
- 右下角区域保留缩放与视角类操作

## 数据与实现约束
1. 自动排列优先复用现有 `nodePositions` 持久化结构，不新增复杂后端表结构。
2. 流程解释优先根据 `workflow.requests`、`workflow.edges`、`inputFields`、`outputFields`、`inputValues` 推导，不新增服务端接口。
3. 保存增强优先复用当前的自动保存与单工作流持久化接口：
   - `saveWorkflowState`
   - `saveWorkflowItem`
   - `saveWorkflowSelection`
4. 请求拖拽优先复用现有 `fetchWorkflowAvailableRequests` 返回的数据，不新增重复请求源。
5. 节点搜索优先在前端基于当前 `selectedWorkflow.requests` 实时过滤，不依赖后端接口。
6. 运行日志需要新增持久化结构和服务端接口，建议单独设计工作流运行日志集合或存储模型。
7. 若后续需要记录解释面板、请求面板、搜索面板、日志面板的展开状态，可先保存在页面本地状态，不强制持久化。

## 验收标准

### 保存能力验收
1. 用户在画布中执行任一编辑动作后，底部操作条能正确显示“未保存 -> 保存中 -> 已保存”的状态流转。
2. 模拟保存失败时，界面能显示失败状态并提供重试入口。
3. 保存失败未恢复前，点击“运行工作流”不会直接执行。

### 自动排列验收
1. 对于线性工作流，自动排列后节点按自上而下主链路整齐分布。
2. 对于包含分支的工作流，同层节点横向展开且不重叠。
3. 对于孤立节点，自动排列后不会压住主链路。
4. 自动排列后的节点位置刷新后能够正常持久化。

### 流程解释验收
1. 已配置工作流时，解释区能展示概览摘要和步骤列表。
2. 步骤列表顺序与工作流主执行链路一致。
3. 点击解释步骤可以定位到对应节点。
4. 当前选中节点时，解释区能同步高亮对应步骤。
5. 空工作流或不完整工作流时，解释区能给出明确提示。

### 请求拖拽验收
1. 点击“请求”入口后，可以看到可用请求列表。
2. 用户能够将请求从列表拖入画布并成功创建节点。
3. 拖入后的节点位置与落点规则一致，并能正常保存。
4. 拖入后节点能继续参与连线、自动排列和运行。

### 节点搜索验收
1. 点击“搜索”入口后，可以根据名称实时搜索当前工作流节点。
2. 点击搜索结果后，画布会定位并选中目标节点。
3. 无结果时能显示明确空态。

### 运行日志验收
1. 每次工作流运行后都会生成一条日志记录。
2. 日志列表能展示运行时间、运行状态和运行耗时。
3. 刷新页面后，历史日志仍可查看。
4. 点击日志后，能看到该次运行中各节点按顺序执行的列表。
5. 点击节点后，能看到该节点的实际入参内容和实际响应内容。
6. 失败运行的日志能够展示错误节点和失败原因。
