# 代码修复执行规划 - 2026-03-26

## 任务主题

本次只规划两个高优先级修复项，待 review 通过后再实施：

1. 修复工作流 `edges` 持久化丢失问题
2. 修复 `/api/proxy` 的 DNS 校验在 ESM 环境下退化的问题

## 背景定位

### 1. 工作流 edges 持久化丢失

现状：
- 前端工作流状态定义包含 `edges`
- 前端页面绘制和交互依赖 `edges`
- 后端 `normalizeWorkflow()` 仅保留 `requests`、`createdAt`、`updatedAt`、`nodePositions`，没有保留 `edges`

直接影响：
- 工作流连线在保存后重新加载会丢失
- 历史数据中即使写入过 `edges`，读取时也会被归一化过程吞掉

涉及位置：
- `src/store/workflowStore.ts`
- `src/pages/WorkflowPage.tsx`
- `server/index.js`

### 2. 代理接口 DNS 校验在 ESM 下退化

现状：
- `/api/proxy` 中使用 `require('dns').lookup(...)`
- 当前项目是 ESM 工程，服务端入口同样以 ESM 方式运行
- 这段校验被包在 `try/catch` 中，异常会被吞掉并继续放行请求

直接影响：
- DNS rebinding 防护可能无法真正执行
- 当前看似存在的安全校验，实际可能只剩 URL/localhost 的浅层检查

涉及位置：
- `server/index.js`

## 修复目标

### 目标 1

保证工作流的 `edges` 字段在以下路径上完整保留：
- 前端 store -> API 请求体
- 服务端归一化 -> MongoDB 存储
- MongoDB 读取 -> API 响应 -> 前端恢复

### 目标 2

保证 `/api/proxy` 的 DNS 校验在当前 ESM 工程下可稳定执行，并且：
- 不再依赖 CommonJS `require`
- 私网/回环地址拦截逻辑仍然有效
- 构建和运行行为可验证

## 非目标

本轮不扩展以下范围，除非 review 时明确追加：

- 不做 `server/index.js` 的大规模模块拆分
- 不重构整个工作流数据模型
- 不全面重写代理层安全策略
- 不顺手处理所有 `admin` 用户名特判
- 不一次性补齐完整后端测试体系

## 执行步骤

### 阶段 0：基线确认

1. 复核工作流 `edges` 的读写链路，确认当前是服务端归一化时丢失，而不是前端未提交。
2. 复核 `/api/proxy` 中 DNS 校验的调用路径，确认 ESM 下的实际退化点和异常吞没路径。
3. 在开始修改前记录影响面，只限定到最小必要文件，避免把这次修复扩大成结构性重构。

### 阶段 1：修复工作流 edges 持久化

1. 在服务端补充 `edges` 的归一化逻辑，新增独立的 edge 归一化步骤。
2. 将 `normalizeWorkflow()` 扩展为显式保留 `edges`，并对非法 edge 做最小清洗。
3. 对历史脏数据做兼容：
   - `edges` 缺失时回退为空数组
   - 结构不合法的 edge 不进入最终持久化结果
4. 检查删除节点后的数据一致性路径，评估是否需要在归一化时顺带过滤引用不存在节点的 edge。
5. 保持前端接口结构不变，不额外修改工作流页面的业务交互，除非验证时发现前端还有协同 bug。

### 阶段 2：修复代理 DNS 校验实现

1. 将 DNS 查询逻辑改为 ESM 兼容的静态导入实现。
2. 把 DNS 查询与私网 IP 判断收敛为清晰的辅助逻辑，避免在路由函数内继续堆叠匿名分支。
3. 保留现有 SSRF 防护主路径：
   - URL 格式校验
   - localhost / 回环地址拦截
   - 私有网段拦截
4. 明确 DNS 查询异常时的策略，并在代码中体现一致行为，而不是依赖大而宽的 `catch` 兜底。
5. 保证修改后不引入新的运行时模块格式错误，也不影响现有超时和响应大小限制。

### 阶段 3：补充验证

1. 构建验证：
   - `pnpm build`
2. 前端回归验证：
   - 重点确认工作流页面导入、保存、刷新后的连线保留
3. 代理接口验证：
   - 合法外部域名请求可以继续通过
   - `localhost` / `127.0.0.1` / 私网地址继续被拒绝
   - DNS 校验逻辑在实际运行中不再因模块格式问题失效
4. 如改动中产生可抽离的纯函数，补最小必要测试；如果本轮不适合补测试，则至少补完整手动验证说明。

## 风险与控制

### 风险 1：修复 edges 时引入旧数据兼容问题

控制方式：
- 归一化逻辑采用向后兼容策略
- 缺失字段一律回退为空结构
- 不改变前端请求格式

### 风险 2：DNS 策略调整导致部分请求行为变化

控制方式：
- 优先做“实现修正”，避免无必要扩大安全策略范围
- 行为变化只限制在真正命中风险的 host/IP 上
- 将 DNS 异常策略单独标注，避免暗改

### 风险 3：在单文件后端里修复时误伤其他接口

控制方式：
- 只在对应 helper 和 `/api/proxy` 路由内做局部修改
- 修改后跑构建并做针对性 smoke check

## 待 Review 确认项

### 确认项 1：无效 edge 的清理范围

建议：
- 最少做结构合法性校验
- 如果 edge 引用的节点不存在，也一并过滤

原因：
- 这类脏数据即使保留下来，前端也没有展示价值，反而可能影响画布渲染

### 确认项 2：DNS 查询失败时的策略

推荐方案：
- 本轮默认“兼容优先”，即修复 ESM 实现，但对纯 DNS 查询失败仍保持当前放行语义

原因：
- 这样能先修掉“校验根本没执行”的问题
- 同时避免把本次修复扩大成外部可感知的策略变更

备选方案：
- “安全优先”，DNS 查询失败直接拒绝代理请求

影响：
- 安全性更强
- 但对部分解析不稳定或特殊域名场景可能产生新阻断

## 执行顺序

推荐按以下顺序实施：

1. 先修工作流 `edges` 持久化
2. 再修 `/api/proxy` DNS 校验
3. 最后统一做构建和针对性验证

原因：
- `edges` 问题是确定的功能正确性 bug，影响面清晰
- DNS 问题涉及安全与兼容性的权衡，放在第二步更便于单独验证行为变化
